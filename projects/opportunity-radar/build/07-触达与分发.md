# 立项 · 第七篇：触达与分发（让用户真正用起来）

> 回应："流程都在网页上，对客户使用麻烦，怎么扩展触达到用户。"
> 核心认知转变：**网页是"运营者的驾驶舱"，不是"用户的入口"。** 用户应该在他们已经在用的地方收到机会。

---

## 一、关键认知：两类人，两个入口

```
┌─────────────────────────────────────────────────────────┐
│  运营者（你）       →  Streamlit 驾驶舱（采集/评估/发布）    │
│  最终用户（订阅者） →  他们已经在用的渠道（不该来网页）       │
└─────────────────────────────────────────────────────────┘
```

> 之前我们只建了"驾驶舱"，缺了"用户入口"。这一篇补上 **分发层（Publisher）**——和采集层（Collector）对称：
> - Collector：把世界的信号**拉进来**。
> - Publisher：把你加工的机会**推出去**，到用户面前。

---

## 二、用户在哪？（按赛道选渠道）

不同赛道的用户，活跃在不同渠道。分发必须跟着用户走：

| 赛道 | 用户主要在哪 | 首选分发渠道 |
|------|------------|------------|
| 加密Web3 | Telegram、X(Twitter)、Discord | **Telegram 频道/Bot** |
| AI工具 | 微信(公众号)、小红书、X | **公众号 + 小红书**（手动）+ 邮件 |
| 开发者开源 | RSS、邮件、X、Telegram | **RSS + 邮件 + Telegram** |
| 通用 | 邮箱 | **邮件 Newsletter** |

> 结论：**没有"一个渠道通吃"。** 但优先级清晰——
> - 能自动化、即时、适合加密/开发者的：**Telegram**。
> - 通用、可沉淀名单的：**邮件**。
> - 国内 AI 受众、需人工排版的：**公众号/小红书**（工具产出"半成品文案"，人工润色发布）。
> - 给极客的：**RSS 输出**（你的机会变成一个别人能订阅的 feed）。

---

## 三、分发层架构设计

```
机会卡片库(opportunities, status=published)
        │
        ▼
   [Publisher 分发器]  ← 与 Collector 对称
        │
        ├── 📄 文件导出   → Markdown / 公众号HTML / 小红书文案（已有简报，增强）
        ├── 📡 RSS 输出   → 生成一个 feed.xml，用户用阅读器订阅
        ├── 🤖 Telegram   → 推送到频道/群（自动、即时）
        ├── 📧 邮件        → 发送给订阅名单（通用、沉淀名单）
        └── 🌐 公开页      → 一个静态页/小站，用户可浏览（可选）
```

### 设计原则
1. **一次加工，多渠道分发**：机会卡片是唯一数据源，各渠道是不同"渲染器"。
2. **渠道可插拔**：和采集器一样，每个渠道一个 Publisher 类，按需启用。
3. **人在回路**：自动渠道（Telegram/RSS）也应支持"人工确认后发布"，避免误发。
4. **合规随行**：每个渠道输出都带免责声明。

---

## 四、分阶段落地（按"投入产出比"排序）

### 阶段 A：零门槛先用起来（立即做）
- **增强文件导出**：在简报基础上，针对渠道出不同格式：
  - 公众号：带小标题、emoji、排版友好的 Markdown/HTML
  - 小红书：短、钩子开头、话题标签、分点
  - Telegram/纯文本：紧凑版
- **RSS 输出**：生成 `feed.xml`，你自己/早期用户可用任意阅读器订阅。
> 价值：你马上能把机会"搬出网页"，复制粘贴或订阅，不用截图。

### 阶段 B：自动触达（验证有受众后）
- **Telegram Bot/频道**：published 的机会一键/定时推送到频道。加密和开发者受众天然在 TG。
- **邮件 Newsletter**：接入邮件服务，发给订阅名单，沉淀可带走的资产。
> 价值：用户在自己的 App 里就收到，零摩擦。

### 阶段 C：规模化（有量之后）
- **公开订阅页/小程序**：用户自助订阅、选赛道、设阈值（如"只推 L4 加密机会"）。
- **个性化推送**：按用户关注的赛道/等级筛选后再推。

---

## 五、本次先实现什么（MVP+）

按阶段 A + 阶段 B 的 Telegram，落地三件：

1. **多格式导出器**（`publishers/exporters.py`）：公众号 / 小红书 / 纯文本 三种渲染。
2. **RSS 输出**（`publishers/rss_out.py`）：把 published 机会生成标准 feed.xml。
3. **Telegram 推送**（`publishers/telegram_pub.py`）：配置 bot token + chat id 即可推送（无配置时跳过，和 AI 一样优雅降级）。
4. UI 的"📰 简报生成"页升级为"📤 分发中心"，集中这些渠道。

> 邮件、公开订阅页放到下一阶段（需要邮件服务/部署，先把零依赖能跑的做了）。

## 六、对"用户真正用起来"的完整路径

```
你在驾驶舱评估好机会 → 标记 published
        ↓
分发中心：选渠道
   ├─ 复制"公众号/小红书文案" → 贴到平台发布（国内AI受众）
   ├─ RSS feed.xml → 极客/早期用户订阅
   └─ 一键推 Telegram 频道 → 加密/开发者受众即时收到
        ↓
用户在自己习惯的地方收到机会 → 不用来你的网页
```

> 这才是"产品被用起来"的样子：**用户感知到的是渠道里的优质机会，而不是一个要登录的网页工具。**
