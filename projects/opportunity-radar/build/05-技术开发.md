# 立项 · 第五步：技术开发

> 立项流程：市场调研 → 商业模式 → 运营 → 产品设计 → **技术开发(本篇)**
> 输入依据：`04-产品设计.md` 的工具系统设计（机会雷达引擎），MVP = 采集①+初筛②+评估③。
> 建议技术栈：**Python 后端 + 轻量 Web 界面 + SQLite 起步**；先本地自用，验证后 Web 化；AI 辅助初筛/起草。

---

## 一、技术目标与原则

### 目标
做出"机会雷达引擎"的 MVP：让你单人能**自动采集信号 → AI辅助初筛 → 框架化评估**，把最耗时的环节自动化，判断留给人。

### 工程原则
1. **小而能跑 > 大而完美**：MVP 先跑通端到端，再逐步加功能。
2. **人机分工**：机器采集/初筛/起草，人做判断/终审（见产品设计铁律）。
3. **数据资产优先**：机会卡片库从第一天就结构化存储，是核心资产。
4. **可演进**：本地脚本 → 轻Web → 可部署应用，架构留好升级路径。
5. **安全合规**：密钥不入库（已配 `.gitignore`），尊重数据源条款和 robots，控制抓取频率。

---

## 二、技术栈选型

| 层 | 选型 | 理由 |
|----|------|------|
| 语言 | **Python 3.11+** | 数据采集、处理、AI 生态最强，适合本项目 |
| 采集 | `httpx` / `feedparser` / `playwright`(必要时) | RSS 用 feedparser，API 用 httpx，动态页用 playwright |
| 数据库 | **SQLite**（起步）→ PostgreSQL（规模后）| 零配置起步，单文件，够用；规模后平滑迁移 |
| ORM/数据 | `SQLModel` 或 `SQLAlchemy` | 类型友好，便于演进 |
| AI 辅助 | 大模型 API（初筛打标、起草）| 你已熟悉 AI；用便宜模型做初筛，好模型做起草 |
| 任务调度 | `APScheduler`（起步）→ cron/系统服务 | 定时采集 |
| Web 界面 | **FastAPI + 简单前端**（起步可用 Streamlit 更快）| 评估工作台需要交互界面 |
| 配置 | `pydantic-settings` + `.env` | 密钥环境变量，不入库 |

> **更快起步的备选**：评估工作台 MVP 可先用 **Streamlit** 几十行搞定交互界面，验证后再用 FastAPI + 前端正式化。建议 MVP 用 Streamlit 抢速度。

## 四、数据库设计（MVP 三张核心表）

```sql
-- 信息源表：你的"生产资料"清单（对应概念篇信息源登记表）
CREATE TABLE sources (
    id           INTEGER PRIMARY KEY,
    name         TEXT NOT NULL,
    type         TEXT,          -- rss/api/web/manual
    url          TEXT,
    track        TEXT,          -- 赛道：AI工具…
    layer        INTEGER,       -- 信息源层级 1-5（①一手…⑤大众）
    value_rating INTEGER,       -- 价值评级 1-3
    enabled      INTEGER DEFAULT 1,
    last_fetched TEXT,
    created_at   TEXT
);

-- 原始信号表：采集进来的未加工信号（收件箱）
CREATE TABLE signals (
    id           INTEGER PRIMARY KEY,
    source_id    INTEGER REFERENCES sources(id),
    raw_title    TEXT,
    raw_content  TEXT,
    url          TEXT,
    hash         TEXT UNIQUE,   -- 去重用
    ai_tags      TEXT,          -- AI 初筛打的标签(JSON)
    ai_relevance REAL,          -- AI 评的相关度 0-1
    status       TEXT DEFAULT 'new',  -- new/filtered_out/promoted
    fetched_at   TEXT
);

-- 机会卡片表：核心资产（对应产品设计的数据模型）
CREATE TABLE opportunities (
    id            INTEGER PRIMARY KEY,
    signal_id     INTEGER REFERENCES signals(id),  -- 来源信号
    title         TEXT NOT NULL,
    track         TEXT,
    dimension     TEXT,          -- 套利维度
    summary       TEXT,
    why_matters   TEXT,
    risks         TEXT,          -- JSON 数组
    fit_for       TEXT,
    half_life     TEXT,          -- 快/可观望/长期
    -- 六维评分（知识库框架）
    score_spread     INTEGER,
    score_capital    INTEGER,
    score_scale      INTEGER,
    score_moat       INTEGER,
    score_risk       INTEGER,
    score_compliance INTEGER,
    judgment      TEXT,          -- 关注/观望/跳过
    judgment_reason TEXT,
    disclosure    TEXT,
    status        TEXT DEFAULT 'draft', -- draft/published/tracking/reviewed
    outcome       TEXT,          -- 战绩追踪：事后填实际结果
    outcome_hit   TEXT,          -- hit/miss/neutral
    channels      TEXT,          -- 已发布渠道(JSON)
    created_at    TEXT,
    published_at  TEXT
);
```

> 设计要点：`signals` 是"原料"，`opportunities` 是"成品"。一条信号经人工评估"晋升"为一张机会卡片。`outcome*` 字段支撑战绩追踪（信任资产）。

## 六、MVP 开发计划（端到端最小闭环）

> 目标：**两周内跑通"采集 → 初筛 → 人工评估 → 存成机会卡片"的最小闭环**，立刻能辅助你的内容生产。

### 里程碑 M0：脚手架（0.5 天）
- [ ] 建 `engine/` 目录、`pyproject.toml`、`.env.example`、`config.py`。
- [ ] `db.py` + `models.py`：建三张表，SQLite 跑起来。
- [ ] `sources_seed.py`：导入一批初始 RSS 源（AI 赛道）。

### 里程碑 M1：采集闭环（2-3 天）
- [ ] `rss_collector.py`：拉取 RSS 源 → 写入 `signals`，带去重(hash)。
- [ ] `scheduler.py`：定时跑采集。
- [ ] 验证：能稳定把多个源的新内容收进收件箱。

### 里程碑 M2：初筛（2-3 天）
- [ ] `filter.py`：关键词/规则初筛（标记明显无关/噪音）。
- [ ] `ai_tagger.py`：调用大模型给信号打标签 + 相关度评分（用便宜模型，控成本）。
- [ ] 验证：收件箱里高相关信号能被自动顶上来，噪音被压下去。

### 里程碑 M3：评估工作台（3-4 天）
- [ ] `ui/app.py`（Streamlit）：
  - 浏览待评估信号（按 AI 相关度排序）。
  - 一键"晋升"为机会卡片，填六维评分 + 判断 + 风险（按知识库框架）。
  - 机会卡片库浏览/编辑。
  - 战绩追踪：给已发布机会回填 outcome。
- [ ] 验证：你能在界面里走完"看信号→评估→存卡片"的完整流程。

### 里程碑 M4：自用试跑（持续）
- [ ] 用它支撑真实的每周内容生产，边用边改。
- [ ] 这也正好对应运营阶段1的"免费跑+攒战绩"。

> **MVP 不做**：加工器(自动生成文案)、分发器(自动发布)、看板。这些放阶段2-3。早期加工分发手工，先验证"采集+评估"这个核心引擎的价值。

## 八、成本与风险（技术层）

- **成本**：服务器（早期本地零成本）+ AI API（初筛用便宜模型，月成本可控在很低）。
- **抓取合规**：尊重 robots/ToS、控频率、不抓登录墙后内容、标注来源。**这点也是 `06-合规` 的延伸。**
- **密钥安全**：全部走 `.env`，已在 `.gitignore` 排除，绝不入库。
- **过度工程风险**：⚠️ 最大的技术风险是"先花一个月写完美系统，却没验证内容有没有人要"。**对策：MVP 极简，先服务内容生产，让工具和内容同步迭代。**

---

## 九、技术开发结论

1. **技术栈**：Python + SQLite + Streamlit(MVP) → FastAPI(后期)，AI 辅助。
2. **MVP = 采集+初筛+评估工作台**，两周跑通最小闭环。
3. **机会卡片库是核心数据资产**，从第一天结构化存储。
4. **人机分工铁律**贯穿：机器提效、人工保判断。
5. **避免过度工程**：工具服务内容，同步迭代，先验证需求再做重。
