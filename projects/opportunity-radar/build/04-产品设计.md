# 立项 · 第四步：产品设计

> 立项流程：市场调研 → 商业模式 → 运营 → **产品设计(本篇)** → 技术开发
> 本篇回答：这个项目到底"产出什么产品"、产品长什么样、信息架构如何、以及支撑它的**内部工具系统**怎么设计（为第五步技术开发铺路）。

---

## 一、产品全景：两类产品

这个项目其实有**两层产品**，别混淆：

```
┌─────────────────────────────────────────────┐
│  对外产品（用户看到的）— 内容产品              │
│  · 机会简报 / 笔记 / 红利手册 / 付费社群        │
│  价值：帮用户省时间、给判断、避坑               │
├─────────────────────────────────────────────┤
│  对内产品（你自己用的）— 工具系统（机会雷达引擎）│
│  · 采集 → 过滤 → 评估 → 加工 → 分发 的工作台    │
│  价值：让你一个人也能高质量高效率持续产出        │
└─────────────────────────────────────────────┘
```

> 关键认知：**对内工具系统是这个项目的"生产线"，也是你的技术能力能转化成护城河的地方。** 它既提升产出效率，又可能在阶段3独立产品化（衔接知识库第二步的"工具产品"方向）。第五步技术开发主要就是做它。

---

## 二、对外内容产品设计

### 产品 1：每周机会简报（核心产品）
- **载体**：公众号（免费）+ 知识星球（付费及时版，阶段3）。
- **结构**：沿用概念篇 `templates/每周简报.md`——速览区(30秒) + 机会详情(卡片) + 上期追踪 + 方法论 + 风险提示 + 免责。
- **差异化**：每个机会都有"判断 + 风险 + 适合谁 + 时效"，不是资讯罗列。

### 产品 2：小红书机会笔记（拉新产品）
- **形态**：单个机会/清单体图文，钩子标题 + 实在内容 + 引导关注。
- **模板化**：固定视觉风格和结构，建立辨识度。

### 产品 3：数字产品（阶段2变现）
- **AI 工具红利手册**：精选+持续更新的"值得用的AI工具+红利+用法"清单。
- **方法论小册**：教用户"怎么自己判断AI工具/机会"（卖认知）。
- **Prompt/工作流模板包**：即用型资产。

### 产品 4：付费社群（阶段3变现）
- 知识星球：及时机会推送 + 深度 + 答疑 + 同好社区 + UGC。

### 内容产品的统一"原子单元"：机会卡片
所有内容产品，底层都是同一个数据结构——**机会卡片**。这非常重要：
> **一次评估加工出一张结构化机会卡片，可以同时复用到简报、笔记、手册、社群。** 这是"一份加工卖多次"的规模化基础，也是工具系统要管理的核心数据。

---

## 三、机会卡片的数据模型（连接内容与工具）

> 这是整个产品的"数据原子"。工具系统围绕它构建。

```
Opportunity（机会）
├── id                    唯一标识
├── title                 机会名称
├── track                 赛道（AI工具/红利…）
├── dimension             套利维度（信息/规模/形态…）
├── summary               一句话是什么
├── source[]              信号来源（含链接、层级①-⑤、发现时间）
├── why_matters           为什么值得关注（价差/机会）
├── risks[]               风险与坑（必填）
├── fit_for               适合谁 / 不适合谁
├── half_life             时效/半衰期（快/可观望/长期）
├── score                 评估打分（六维，来自知识库框架）
│   ├── net_spread        净价差
│   ├── capital_eff       资金效率
│   ├── scalability       可规模化
│   ├── moat              护城河
│   ├── risk              风险可控
│   └── compliance        合规匹配
├── judgment              我的判断（关注/观望/跳过 + 理由）
├── disclosure            利益披露
├── status                状态（待评估/已发布/追踪中/已复盘）
├── outcome               实际结果（用于战绩追踪，事后填）
├── created_at / published_at
└── channels[]            已发布到哪些渠道
```

> 这个模型同时服务：①内容生产（填完即可生成各渠道内容）②战绩追踪（outcome 字段）③数据沉淀（历史机会库=资产）。

## 四、对内工具系统设计（机会雷达引擎）

> 这是第五步技术开发的主体。先在产品层定义它"该有什么功能"，技术篇再定"怎么实现"。

### 系统目标
让你（单人）能够：**高效采集 → 智能初筛 → 框架化评估 → 快速加工 → 多渠道分发 → 自动追踪战绩**，把每周几十小时的手工活压缩到可持续的工作量。

### 功能模块（MVP → 完整）

| 模块 | 功能 | MVP? | 说明 |
|------|------|:----:|------|
| **① 采集器** | 聚合 RSS/API/网页/社区信号到统一收件箱 | ⭐MVP | 让信息自动来 |
| **② 初筛器** | 去重 + 关键词/规则过滤 + AI 辅助打标 | ⭐MVP | 砍掉 90% 噪音 |
| **③ 评估工作台** | 按知识库六维框架给机会打分、记判断 | ⭐MVP | 人工为主，核心判断 |
| **④ 加工器** | 机会卡片 → 一键生成各渠道文案（AI辅助起草）| 阶段2 | 提效，人工终审 |
| **⑤ 分发器** | 推送到公众号/星球/群（或半自动）| 阶段2-3 | 降低交付成本 |
| **⑥ 战绩追踪** | 自动/半自动跟踪机会后续结果 | 阶段2 | 信任资产 |
| **⑦ 数据看板** | 机会库、信息源质量、内容数据统计 | 阶段3 | 决策支持 |

> **MVP 只做 ①②③**：先解决"采集+初筛+评估"这个最耗时、最体现专业度的环节。加工和分发早期可手工。

### 人机分工原则（关键设计哲学）
```
机器做：采集、去重、初筛、打标、起草初稿、数据统计  →  降本提效
人做：  最终过滤判断、机会评估、观点、终审、战绩定性  →  保证质量和信任
```
> ⚠️ 铁律（来自 `06-合规与信任`）：**"判断"绝不能完全交给机器。** 你卖的就是判断和信任，AI 只做辅助。一旦 AI 起草的内容没人把关就发出去且出错，信誉崩塌。

## 五、信息架构（数据如何流动）

```
[信息源矩阵]
   │ 采集器(①)
   ▼
[原始信号收件箱] ──初筛器(②)──> [待评估池]
   │                                │ 评估工作台(③)
   │                                ▼
   │                          [机会卡片库] ◄── 核心数据资产
   │                                │
   │                  ┌─────────────┼─────────────┐
   │              加工器(④)      分发器(⑤)      战绩追踪(⑥)
   │                  ▼             ▼             ▼
   │            [各渠道文案]   [公众号/星球/群]  [战绩记录]
   │                                              │
   └──────────────── 数据看板(⑦) ◄───────────────┘
```

> 机会卡片库是中心。所有上游（采集/评估）汇入它，所有下游（内容/分发/战绩）从它流出。**这个库本身就是越积越值钱的资产。**

## 六、用户体验设计要点（对外）

- **省时间感**：简报的"30秒速览区"让忙人秒读；详情给认真看的人。
- **信任感**：战绩追踪公开可查；风险提示醒目；不夸大。
- **行动力**：每个机会明确"该不该关注、下一步看什么、时效多久"。
- **学习感**：方法论板块让用户"学会织网"，强化粘性和差异化。
- **辨识度**：统一的视觉/结构模板，形成品牌记忆。

---

## 七、产品设计结论

1. **两层产品**：对外内容产品（简报/笔记/手册/社群）+ 对内工具系统（机会雷达引擎）。
2. **机会卡片是数据原子**：一次加工、多渠道复用，是规模化和战绩追踪的基础。
3. **工具系统 MVP = 采集+初筛+评估**（最耗时、最体现专业度的环节）。
4. **人机分工铁律**：机器提效，人工保判断和信任。
5. **机会卡片库是核心资产**：越积越值钱，也是潜在的独立产品。

## 八、待你确认的决策点（进入技术开发前）

> 请拍板（或"按建议来"）：

1. **MVP 工具范围**：认可"先做 采集+初筛+评估 工作台（①②③），加工分发先手工"吗？
2. **技术栈倾向**：你有技术能力——倾向 Python（数据/AI 生态强、适合采集处理）还是 Node/TS（全栈、前端友好）？我建议 **Python 后端（采集+处理+AI）+ 轻量 Web 界面**。
3. **形态**：工具系统先做**本地/自用脚本+轻Web**，还是直接做**可部署的Web应用**？我建议先本地自用跑通，验证后再 Web 化。
4. **AI 辅助程度**：初筛和起草用 AI（调用大模型 API）辅助，你认可吗？（会有少量 API 成本）
